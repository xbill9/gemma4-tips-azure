# 🤖 Gemini Workspace Context: GPU vLLM DevOps Agent

This context guide summarizes the configuration, optimal serving parameters, and capabilities of the self-hosted **Gemma 4 DevOps/SRE Agent** running on **Cloud Run GPU** (NVIDIA L4).

> [!IMPORTANT]
> - Never suggest or deploy the 26B model configuration.
> - Always utilize the 31B model quantized at 4-bit (NVFP4 format) loaded on NVIDIA L4 GPU (`nvidia/Gemma-4-31B-IT-NVFP4`).

---

## ⚙️ Active Environment Configuration

This agent targets Google Cloud Platform (GCP) deployments utilizing:
- **Project ID**: `aisprint-491218` (configurable via `GOOGLE_CLOUD_PROJECT`)
- **Region**: `us-east4` (configurable via `GOOGLE_CLOUD_LOCATION`)
- **Default Model**: `nvidia/Gemma-4-31B-IT-NVFP4` (configurable via `MODEL_NAME` / `MODEL_PATH`)
- **GPU Accelerator**: NVIDIA L4 GPU (1 unit) on Cloud Run Gen2
- **Default Cloud Run Service Name**: `gpu-31b-l4-devops-agent`

## 🚀 Recommended vLLM Configuration for Gemma 4

To achieve stable and performant Gemma 4 serving with tool/function calling support on NVIDIA L4 GPU, use the following container startup arguments:

```yaml
args:
  - --model
  - /mnt/models/nvidia/Gemma-4-31B-IT-NVFP4
  - --dtype
  - float16
  - --quantization
  - nvfp4
  - --cpu-offload-gb
  - "14"
  - --safetensors-load-strategy
  - lazy
  - --max-model-len
  - "512"
  - --disable-chunked-mm-input
  - --gpu-memory-utilization
  - "0.90"
  - --kv-cache-dtype
  - fp8
  - --tensor-parallel-size
  - "1"
  - --max-num-seqs
  - "4"
  - --enforce-eager
  - --enable-chunked-prefill
  - --max-num-batched-tokens
  - "4096"
  - --enable-auto-tool-choice
  - --tool-call-parser
  - gemma4
  - --reasoning-parser
  - gemma4
  - --async-scheduling
  - --limit-mm-per-prompt
  - '{"image": 0}'
  - --trust-remote-code
  - --host
  - 0.0.0.0
  - --port
  - "8000"
```

### Key Parameters & Optimizations Explained
*   **`--quantization nvfp4`**: Nvidia 4-bit floating point quantization to fit the massive 31B parameter model on a single 24GB L4 GPU.
*   **`--cpu-offload-gb 14` (or 11)**: Allocates 11-14 GiB of system RAM for offloading weights, ensuring the 31B model can start without running out of GPU memory.
*   **`--max-model-len 512`**: Constrains the maximum sequence length to 512 tokens to conserve memory footprint on L4.
*   **`--max-num-seqs 4`**: Limits concurrent requests to 4 sequences, reducing the active memory usage.
*   **`--enforce-eager`**: Disables CUDA graphs to save memory, which is critical when offloading weights to CPU.
*   **`--gpu-memory-utilization 0.90`**: Allocates 90% of VRAM to vLLM's KV cache.
*   **`--kv-cache-dtype fp8`**: Uses FP8 for KV cache, reducing memory consumption and increasing capacity.
*   **`--tool-call-parser gemma4` & `--reasoning-parser gemma4`**: Essential settings for correct parsing of tool calls and structured reasoning steps generated by Gemma 4.
*   **Optimized Environment Variables**:
    - `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`: Reduces memory fragmentation in PyTorch.
    - `VLLM_USE_V1=0`: Disables the v1 engine to maintain compatibility with specific features.
    - `MALLOC_TRIM_THRESHOLD_=65536`: Dynamically returns memory back to the OS.
*   **GCS FUSE Mount Options**:
    - `mount-options=uid=1001;gid=1001;stat-cache-ttl=3600s;type-cache-ttl=3600s;max-conns-per-host=100`: Speed up model weights reading and mounting from GCS.

## 🧰 Key SRE & DevOps Capabilities

This agent exposes several tool categories via the Model Context Protocol (MCP):
- **Deployment & Scaling:** `deploy_vllm`, `destroy_vllm`, `status_vllm`, `update_vllm_scaling`, `get_vllm_deployment_config`, `get_vllm_gpu_deployment_config`, `check_gpu_quotas`, `get_vllm_endpoint`.
- **Model Transfer & Secret Management:** `list_vertex_models`, `list_bucket_models`, `save_hf_token`, `get_vertex_ai_model_copy_instructions`, `get_huggingface_model_copy_instructions`, `get_huggingfacehub_download_path`.
- **System Monitoring & Health:** `get_system_status`, `get_endpoint`, `get_model_details`, `verify_model_health`.
- **Performance Benchmarking:** `run_benchmark` (supports custom concurrency and request count sweeps).
- **Diagnostics & SRE Remediation:** `query_gemma4`, `query_gemma4_with_stats`, `query_vllm`, `analyze_cloud_logging`, `analyze_gpu_logs`, `suggest_sre_remediation`, `get_help`.
